# MatchMaker

**Languages**: [English](README.md) · 简体中文

🔗 **在线访问**：**[CaiZhou2.github.io/MatchMaker](https://CaiZhou2.github.io/MatchMaker/)** —— 直接在手机浏览器打开，添加到主屏幕，开始用。

一个手机友好的 Progressive Web App，用于安排每周的团队杯赛。MatchMaker 维护持久化的选手名单，根据历史胜率自动抽取平衡的队伍，智能推荐合适的赛制，运行比赛，并跨周记录积分和消费 —— 完全在手机浏览器里运行，无需安装，无需账号。

> **仅限非商业用途。** 详见 [LICENSE](LICENSE)。

> ⭐ **如果 MatchMaker 对你的小组有帮助，恳请给项目点一个 star** —— 这是帮助其他组织者发现这个工具最简单的方式，对你没有任何成本。谢谢！

📖 **只是想用它办比赛？** 不用看下面的开发者文档，直接读 [**组织者快速上手指南**](docs/ORGANIZER_GUIDE.zh-CN.md) —— 写给非技术用户的，10 分钟从零跑完第一场比赛。

## 功能特性

- **每周杯赛流程**：选择参赛者 → 自动组队 → 比赛 → 录入结果 → 提交
- **平衡组队**：按胜率排名前 `N` 的选手作为队长（每队一名），其余选手随机轮询分配
- **5 种比赛模式**让用户每场自选：**自动**（推荐 —— 依次尝试小组赛+淘汰、循环赛、纯友谊）、**小组赛 + 淘汰赛**、**纯小组赛（循环赛）**、**纯淘汰赛**（2 的幂次队伍数，标准种子）、**纯友谊赛**（公平随机配对 —— 不计积分，但胜率 / 胜平负 / 对位胜率仍会更新）
- **淘汰赛友谊场**：为淘汰队伍保留一个场地用于自由比赛，避免无聊
- **积分榜 + 胜率榜**：`胜 = 3 分`，`平 = 1 分`，`负 = 0 分`
- **消费追踪**：本周总花费自动均摊到参赛者；每人累计消费；双重确认清零 + 下次比赛前可恢复
- **比赛安排分享**：一键复制文本，可粘贴到微信 / Telegram / 其他聊天
- **历史归档**：每场完成的比赛都保存完整细节（队伍、结果、积分/消费变化、姓名快照）
- **数据导入/导出**：完整 JSON 备份/恢复
- **离线使用**：安装为 PWA 后支持离线运行
- **多语言**：自动检测浏览器语言（简体中文或美式英语），支持手动切换
- **无后端**：所有数据本地存储在 `localStorage`

## 快速开始

### 本地运行

只需要一个静态 HTTP 服务器 —— 无需构建，无依赖。

```bash
git clone https://github.com/<your-user>/MatchMaker.git
cd MatchMaker/web
python3 -m http.server 8080
# 浏览器打开 http://localhost:8080
```

任何静态服务器都可以（`python3 -m http.server`、`npx serve`、nginx、caddy、GitHub Pages、Netlify 等）。

### 安装到手机（PWA）

MatchMaker 是一个渐进式 Web 应用（PWA），可以从任何现代手机浏览器"安装"到主屏幕，像原生应用一样使用 —— 离线、全屏、带图标。

**最简单的方式 —— 直接用线上版本：** 在手机浏览器打开 **<https://CaiZhou2.github.io/MatchMaker/>**，然后跳到下面的安装步骤。不需要自己部署。

## 使用流程

### 1. 建立选手数据库

- 主页 → **选手数据库** → 输入姓名 → **添加**。
- 每个新选手只需添加一次，数据会跨周累积。

### 2. 开始本周比赛

- 主页 → **开始本周比赛**。
- 勾选本周参赛者。也可以在这里快速添加新选手。
- 设置：
  - **每队人数**
  - **场地数量**
  - **每场时长**（分钟）
  - **总时间限制**（分钟）
  - **本周消费**（¥）—— 提交时均摊到参赛者
- 点 **生成队伍**。

### 3. 审核队伍

- 队长（按胜率排序）带 👑 标记。
- 点 **🔄 重新随机** 再抽一次。
- 点 **🔀 手动调整** 交换两名选手。
- 底部的赛制预览显示本次会打什么：小组赛+淘汰赛、循环赛、单败淘汰赛、或纯友谊赛（取决于在设置页选的模式）。
- 点 **📋 复制比赛安排** 分享到聊天群。
- 点 **开始比赛** 锁定方案。

### 4. 运行比赛

- 每个时段显示各场地的比赛。
- 点击 **A胜 / 平局 / B胜** 录入结果。
- 所有正赛都录入后，完成按钮会出现。
- 点 **完成比赛 & 更新积分** → 确认 → 积分和消费写入数据库。

### 5. 查看和管理

- 主页 **排行榜**（积分榜 / 胜率榜 / 消费榜 三个 Tab）
- **选手数据库** 查看/删除个人数据
- **历史记录** 展开查看过往比赛细节
- **消费统计** 卡片：总消费 + 清零（双重确认）+ 恢复（下次比赛提交前有效）
- **数据** → **导出 JSON** / **导入 JSON**

### 数据模型

localStorage 里的键 `matchmaker-data-v1`：

```jsonc
{
  "players": {
    "p_xxx": {
      "id": "p_xxx",
      "name": "张三",
      "points": 30, "wins": 10, "draws": 0, "losses": 0,
      "events": 5,
      "totalSpent": 160.00
    }
  },
  "currentEvent": null | { ... },         // 进行中的比赛
  "history": [ { ... } ],                 // 已完成的比赛（带完整快照）
  "expenseBackup": null | { "p_xxx": 40 } // 清零后的临时备份

```

### 算法要点

> **友谊第一。** MatchMaker 里所有的组队和配对算法都为一个目标
> 服务：让比赛**势均力敌**——不是为了"评出谁最强"，而是让水平
> 相近的两队（用胜率作为简单的 Elo 代理）站在球网两边，让所有
> 人玩得开心，避免一边倒的碾压局。当条件不允许这样做时（人少、
> 水平差距大），算法会优雅地降级到"每个人尽量打同样多场，没人
> 干坐着无聊"，但**永远不会**降级到"最强的玩家拿最多分"。

#### 实力评估：累计胜率

整个项目的"实力"信号就是每个选手在持久化数据库里的累计胜率
——`wins / (wins + draws + losses)`，排序的 tiebreaker 用比赛
场次。这是一个故意保持简单透明的 Elo 代理。我们不维护真正的
Elo 评分，因为小群体周赛的场景下，简单信号已经够用，而且用户
能直观理解。

#### `formBalancedTeams` —— 固定周队伍（杯赛模式用）

所有杯赛模式（小组赛+淘汰、循环赛、单败淘汰）都用这个函数。
按照胜率排名后形成 `T = floor(N / 每队人数)` 支队伍，分两阶段：

1. **蛇形抽签阶段**，覆盖排名前半 `0..⌊N/2⌋ − 1`。蜿蜒走位
   （fantasy draft 风格）：
   - 第 0 轮（正向）：rank 0 → T0, rank 1 → T1, ..., rank T−1 → T_{T−1}
   - 第 1 轮（反向）：rank T → T_{T−1}, rank T+1 → T_{T−2}, ...
   - 第 2 轮（正向）：...

   这样最高胜率的"队长"会被放在 `team[i].players[0]`（所以
   显示和手动调整仍然认得出"队长"），然后剩余的前半玩家被
   分配得让**前半排名之和在各队之间尽量相等**。12 人 × 4 人
   一队的例子：

   ```
   T0 抽到：ranks {0, 5}  → 排名和 5
   T1 抽到：ranks {1, 4}  → 排名和 5
   T2 抽到：ranks {2, 3}  → 排名和 5  ← 完美平衡
   ```

2. **随机填充阶段**，覆盖排名后半 `⌊N/2⌋..T·每队人数`。
   shuffle 后轮询分配到还有空位的队伍。这一档随机分配是 OK 的
   —— 这些都是低胜率玩家，就算抽得"运气不好"也不会造成大的
   失衡，因为前半已经被蛇形稳定住了。

`teamSize = 2`（最常见的场景）下蛇形阶段只填前 `T` 个队长，后
半完全随机 —— 跟历史上的"队长 + 1 个随机"规则完全一致。蛇形
主要在每队人数变大时（3、4、5 人）发挥作用，那时纯随机填充
`2T+` 个非队长玩家容易造出严重失衡的队伍。

旁观者：如果 `N % 每队人数 > 0`，剩下的玩家就当"旁观者"坐场
外。用户可以在设置页调整人员，凑成整除。

#### `planRoundRobin` —— 循环赛（Berger 法）

Berger / 圆周法：`T − 1` 轮（T 偶数）或 `T` 轮（T 奇数，每轮
有一队轮空）。每一轮把场地填满并行进行。每轮 `floor(T/2)` 场
比赛，slot 总数随队伍数增长。

#### `planGroupsKnockout` —— 小组赛 + 淘汰赛

先尝试 4 人一组，再退化到 3 人一组，选时间预算下可行的那种。
每组前 2 名晋级。晋级队伍构成 2 的幂次淘汰表（如果不是 2 的
幂次，从最低种子开始截掉多余的）。淘汰阶段，**只要场地有富余
（`numCourts > 当前轮所需场数`），就保留一个场地给被淘汰队伍
打友谊赛**，让他们不用干坐着无聊。

#### `planPureKnockout` —— 纯单败淘汰

标准的网球种子排位（`seedBracket()`）：1 号种子和 2 号种子分
在两个半区，只能在决赛相遇。**要求队伍数必须是 2 的幂次**
（2 / 4 / 8 / 16 / 32）。非 2 的幂次会被显式拒绝，让用户改模式
而不是引入难看的轮空结构。淘汰阶段同样为被淘汰队伍保留友谊场。

#### `planRandomFairFallback` / `planFriendly` —— 基于模板的逐场配对

算法上最复杂的规划器，被显式的纯友谊赛模式和 Auto 模式的最后
回退路径共用。这里**没有固定周队伍**——每场比赛都有自己临时
组成的人员和分队。

**面向用户的规则**（保持"友谊第一"的口径）：

1. 按累计胜率排序。前 50% = STRONG 池，后 50% = WEAK 池。
2. 每场比赛从 `每队人数 + 1` 个模板中抽一个，参数 `k` 表示
   "每队的强者人数"：
   - `k = 0` → 全弱 vs 全弱（`WW vs WW`）
   - `k = 每队人数` → 全强 vs 全强（`SS vs SS`）
   - 中间的 `k` → 混合（`SW vs SW`、`SSW vs SSW` 等等）

   `每队人数 = 2` 时正好是用户要求的"三个模板"：
   `WW-WW / SW-SW / SS-SS`。每队人数变大时模板列表自然延伸。
3. 选中的模板决定从两个池子各抽多少强、多少弱。如果某个池子
   人不够，从另一个池子补足缺额。
4. 比赛内部：强者轮询分配到各队，弱者按蛇形分配，最终每队
   都拿到正确的 `k` 强 + `(每队人数 − k)` 弱的组合。

**在基本模板抽签之上还有三层精修**：

a. **池子平衡偏置**（参与公平）。在抽模板之前，先比较强池和
   弱池的平均出场次数。如果不相等，**强制选最极端的纠偏模板**
   —— 当强池被冷落时选 `k = 每队人数`（全强），弱池被冷落时
   选 `k = 0`（全弱）。中间模板没法真的缩小参与度差距，因为
   它们对两个池子贡献相等；只有极端模板能纠偏。这样在数学
   允许的情况下能让所有人的出场次数完全相等。

b. **不重复跟踪**（用户的"尽量不要出现 已经在同一个场子出现
   过的相同的人 重新组合后出现在另一场" 规则）。本场赛事产生
   过的每一个 cohort 都被记到 `allCohortKeys` 里（按选手 ID
   排序的字符串作为 key）。挑新 cohort 时，对当前模板重试几
   次（每池贪心选择器有随机 tiebreaker，重试能产生不同的 SW
   组合），第一个 key 不在已用集合里的 cohort 胜出。SS-SS 和
   WW-WW 在小阵容下各自只有一种 cohort，所以每个最多只能用
   一次，之后必须切换到 SW。如果连 SW 也产生不了新的 cohort，
   算法接受被迫的重复 —— 这是数学上无法避免的。

c. **跨池回退**（用户的"群体不够的话再从剩下的人中随机抽" 规则）。
   如果选中的池子不够填模板（比如某 slot 第二个场地时，强池
   可能已经被部分用掉），缺额从另一个池子补足。这里的排除
   逻辑要小心：`pickFromPool` 必须把"当前 cohort 已选的玩家"
   也排除掉（不只是排除"本 slot 已用"的玩家），否则跨池回退
   可能会把已经选过的人再选一次，产生重复玩家比赛。这是个
   真实存在过的 bug —— 见 `tests/fallback.test.js` 的
   "no match has duplicate players" 测试。

**为什么用模板而不是纯贪心蛇形？** 之前试过的"贪心选 4 个共
现次数最低的玩家，按胜率蛇形分队"方法，倾向于让同水平的玩家
每场都聚在一起 —— 因为 game count 相同时，胜率排序就主导了。
模板方法显式地循环 `WW / SW / SS` 三种模式，所以一场赛事下来
用户能看到**三种形态都出现** —— 强强对决、弱弱拉锯战、混合
野球场 —— 而不是"前 4 强永远相互打"。

**为什么这仍然是"友谊第一"？** 一场比赛内部，模板结构保证
两支队伍的强弱构成相同，所以两边天然对称。池子平衡偏置保证
所有人的出场次数差不多。不重复跟踪保证同样的 4 个人不会被
卡在一起反复打。综合下来：每场比赛感觉势均力敌，整场赛事下
来每个人都和很多不同的对手玩过。

#### 友谊赛的结果记账

`planFriendly` 是 `planRandomFairFallback` 的薄包装，把每场都
标记为 `kind: 'friendly'`。`commitEvent` 的 delta 累加器对
友谊赛调用 `accumulateDelta(..., countPoints=false)`：胜 / 平 /
负仍然会被记录（所以胜率榜、对位胜率、选手详情统计都会反映
友谊赛结果），但**不发放任何比赛积分** —— 积分榜完全不动。
同样的规则也作用于 `getHeadToHead`、历史详情视图、比赛结束
总结。

#### 小组赛排名 tiebreaker

`computeGroupTable` 按 **积分 → 净得分 → 确定性随机** 排序。
"随机" 是 `(team.id, event.date, groupIdx)` 的稳定 djb2 hash，
所以每次读取都得到相同的结果（重渲染时不会闪），但又不会退
化为"低 team index 优先"（那等于没有 tiebreaker）。

当一个小组完成（所有小组赛都已录入）并且排名中有任何并列位置
时，比赛页面会通过 `maybeShowTiebreakerNotices()` 弹一个一次
性提示，告诉用户这个并列是怎么解决的（按净得分排序，或者净
得分也相同时随机）。已经显示过的提示通过 `ev.tiebreakerNoticesShown`
持久化跟踪，所以后面如果用户改了某场比赛的结果，提示也不会
重复弹出。

#### 小组完成度门控（占位符解析）

`planGroupsKnockout` 的淘汰赛比赛在生成时使用占位符引用，比如
`"G1-1"`（小组 1 第 1 名）和 `"KR1-M2-W"`（淘汰赛第 1 轮第
2 场的胜者）。这些在渲染时通过 `resolvePlaceholder` 解析。
**G 分支只有当小组的所有比赛都已经录入结果之后才会解析**
（`isGroupComplete` 门控）—— 否则排名表会是一张未打完的表，
没打过的队伍都并列 0 分，"第 1 名"实际上就是 `teams[0]`。
门控生效之前，比赛页面显示的是人类可读的占位符标签
`小组1第1 / Group 1 #1`，不会暴露假的队伍名。

#### `planByMode` 模式分发

单一入口，根据用户在设置页选的模式路由到对应的规划器：

- `auto` —— 先调用 `recommendFormat`（依次尝试小组赛+淘汰、
  循环赛），如果都装不下时间预算，最终回退到 `planFriendly`。
  **永远能给出可玩的方案**，哪怕预算很紧。
- `groups-knockout`、`round-robin`、`knockout`、`friendly` ——
  显式模式。如果选的模式不可行，UI 会弹出具体原因并阻止下一
  步（不会偷偷切换）。用户必须改模式或者改参数（加时间、加
  场地、减人数）。
